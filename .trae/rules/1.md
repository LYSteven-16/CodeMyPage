# CodeMyPage - Cline Rules

## 语言
- 始终用中文回复

## 项目概述
- 纯 TypeScript 项目（无框架依赖），原生 DOM 操作
- 可视化拖拽式网页编辑器，部署到 GitHub Pages（纯静态）
- 技术栈：TypeScript + Vite + UnoCSS
- 引擎：`@component-chemistry/atom-engine`（BeakerManager → 分子包渲染快照）

---

## 🛠️ 技能系统（Skills）

项目目录下有 `.claude/skills/` 技能文件，**当你需要执行特定任务时，主动调用对应技能**。

### 技能调用方式
当你需要使用某个技能时，先读取对应的技能文件理解其流程，然后按技能定义的步骤执行。

### 可用技能

| 技能 | 触发词 | 用途 |
|------|--------|------|
| `verify` | "验证""verify""测试改动" | 验证代码改动是否达到预期 |
| `simplify` | "简化""review""整理" | 代码审查：复用性/质量/安全 |
| `review` | "review""审查""分析改动" | 深度代码审查，输出改进建议 |
| `commit` | "commit""提交""git 提交" | 规范的 Git Commit 流程 |
| `debug` | "debug""调试""有问题""报错" | 定位问题根因 |
| `batch` | "批量""同时做""很多地方要改" | 大任务拆解并行处理 |

### 技能使用优先级
1. **有明确验证需求** → 优先用 `verify`（如改完功能后）
2. **不确定代码质量** → 用 `simplify` 做快速审查
3. **PR 或重要改动** → 用 `review` 做深度审查
4. **出问题/报错** → 直接用 `debug` 诊断

## 📋 标准工作流程

### 流程一：功能改动
```
1. 用户提出需求
2. → 读取相关源码，理解当前实现
3. → 制定计划（输出计划，等用户确认）
4. → 用户确认后执行
5. → 用 verify 技能验证改动
6. → 用 simplify 技能做代码审查
7. → 汇报结果
```

### 流程二：问题修复
```
1. 用户描述问题
2. → 用 debug 技能诊断根因
3. → 定位到具体文件和行
4. → 分析根因，输出报告
5. → 用户确认后再修复
6. → 用 verify 技能验证修复
```

### 流程三：代码审查
```
1. 用户要求审查
2. → 用 simplify 或 review 技能
3. → 输出结构化报告
4. → 用户确认要修复哪些问题
5. → 逐个修复
```

### 流程四：Git 提交
```
1. 用户要求 commit
2. → 用 commit 技能执行
3. → 先展示 commit message，等用户确认
4. → 确认后执行提交
```

## 工作流程（必须遵守）

### 第一步：分析
- 读取相关源码，理解当前实现
- 确认根因（不要猜，要从代码中找证据）

### 第二步：规划（先输出计划，等用户确认再动手）
输出格式：

```
## 目标
（一句话说清楚要做什么）

## 根因
（从代码中找到的根本原因，附文件路径和行号）

## 实施步骤
1. 改文件 A：具体改什么，为什么
2. 改文件 B：具体改什么，为什么
...

## 验收标准
（可验证的标准，不是"应该可以"）
```

### 第三步：执行
- 按确认后的计划逐步实施
- 每改一个文件，说明改了什么
- **改完后主动用 `verify` 技能验证**

### 第四步：验证
- 按验收标准逐条验证
- 运行 `npm run build` 确保无 TS/lint 错误
- 告诉我结果：✅ 达标 / ❌ 哪条没达标

## 编辑原则
- **只修改需要改变的部分**，不要重写整个文件
- 用 edit 工具精确替换（提供 old_string 和 new_string）
- 修改前先读取相关文件（至少 2-3 个），理解上下文再动手
- 遵循项目现有代码风格，不要引入新架构

---

## 代码风格
- TypeScript 严格模式，所有函数要有类型注解
- 变量/函数：camelCase，类/接口：PascalCase，文件名：kebab-case
- 字符串用单引号，语句末尾加分号，缩进 2 空格
- 用 ES Module（import/export），不用 require

---

## 组件系统
- 组件定义在 `src/components/*.json`，格式：type、name、icon、defaultWidth、defaultHeight、molecule
- molecule 是 AtomEngine 格式，包含 atoms 数组
- 新增组件只需添加 JSON 文件，component-registry.ts 自动扫描注册
- **不要为每个组件单独创建 .ts 文件**

---

## 技术约束
- ❌ 不能使用 Node.js API（fs、path 等）
- ❌ 不能有后端请求（GitHub Pages 只支持静态资源）
- ✅ 数据持久化用 localStorage
- ✅ 文件操作用浏览器 API（FileReader、Blob）

---

## 状态管理
- 状态集中放在 `state.ts`，通过 `getState()` / `setState()` 访问
- 状态变更通过 `events.ts` 事件系统通知其他模块
