# Verify 技能 — 验证代码改动

当你需要验证一个功能改动是否正确时使用。

## 使用方式

用户说这些词时触发：
- "验证一下"
- "verify"
- "测试改动"
- "跑一下看看"

## 执行步骤

### 第一步：确认改动内容
运行 `git diff`（或 `git diff --cached`）查看所有变更。
如果不在 git 管理下，用 `git status` 或直接列出最近修改的文件。

### 第二步：制定验证计划
根据改动内容，确定：
1. **功能验证**：需要手动测试什么？（输入什么，预期输出是什么）
2. **构建验证**：运行 `npm run build` 或 `tsc --noEmit` 确保无编译错误
3. **类型检查**：`npm run type-check` 或 tsc（如果有）

### 第三步：执行验证

#### 构建验证（必做）
```bash
npm run build
```

#### 功能验证（根据改动选择）
- 如果改了 UI：描述如何在浏览器中测试
- 如果改了逻辑：列出测试用例和预期结果
- 如果改了组件：确认组件 JSON 格式正确

#### 如果有测试
```bash
npm test
```

### 第四步：输出报告

```
## 验证报告

### 改动摘要
（简述改了什么）

### 验证结果

| 验证项 | 结果 | 说明 |
|--------|------|------|
| 构建 | ✅/❌ | 错误信息（如果有）|
| 类型检查 | ✅/❌ | 错误信息（如果有）|
| 功能测试 | ✅/❌ | 描述测试结果 |

### 结论
✅ 验证通过 / ❌ 存在问题，需要修复
```

## 常见问题处理

- **构建失败**：读取错误信息，定位到具体文件和行，修复后再验证
- **类型错误**：先修复类型问题，再重新构建
- **功能不符**：描述实际行为和预期行为的差异，给出修复建议
