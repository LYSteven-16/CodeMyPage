# Simplify 技能 — 代码审查与清理

当用户要求审查代码或说"简化一下""review""整理整理"时触发。

## 执行步骤

### 第一步：获取改动

```bash
git diff
```

如果文件未纳入 git，列出用户提到的或最近编辑的文件。

### 第二步：三个维度的审查

#### 维度 1：复用性审查
1. **搜索是否有现成工具**：
   - 找类似的模式：类似的字符串处理、路径操作、环境判断
   - 常见可复用位置：`src/utils/`、`src/helpers/`、`src/shared/`
2. **标记重复代码**：
   - 搜索相似的函数或代码块
   - 建议提取为共享函数

#### 维度 2：质量审查
1. **冗余状态**：
   - 是否有重复的 state、缓存值、可以用直接计算替代的 observer
   - 是否有不必要的参数蔓延
2. **复制粘贴变体**：
   - 找近似重复的代码块
   - 建议统一为共享抽象
3. **代码组织**：
   - 函数是否过大（>50行建议拆分）
   - 导入是否混乱

#### 维度 3：安全性审查
1. **硬编码密钥**：
   ```bash
   grep -rn "password\|api_key\|secret\|token\|key.*=.*['\"]" src/
   ```
2. **残留调试代码**：
   ```bash
   grep -rn "console\.log\|debugger\|TODO.*fixme\|FIXME" src/
   ```
3. **危险操作**：
   - `eval()`、`new Function()`
   - 动态代码执行

### 第三步：输出报告

```
## Simplify 审查报告

### 复用性问题
- ❌ 文件 A：第 N 行，重复了 src/utils/helper.ts 的功能
  → 建议：改用 helper.formatXxx()

### 质量问题
- ⚠️ 文件 B：函数过长（80行），建议拆分为...
- ⚠️ 文件 C：有重复的 state 定义

### 安全问题
- ❌ 文件 D：硬编码了 API Key（必须移除）

### 可改进项
- 💡 文件 E：可以用更简洁的写法...

### 总体评价
（简洁 / 有问题需修复）
```

## 注意
- 如果没有发现问题，直接说"未发现明显问题"
- 不要过度挑剔格式问题，聚焦逻辑和架构
