# Review 技能 — 代码审查

当用户说"review""审查""看看这个 PR""分析一下改动"时触发。

## 执行步骤

### 第一步：获取改动

```bash
git diff HEAD
```

或者如果有 PR：

```bash
gh pr view <PR号>
gh pr diff <PR号>
```

### 第二步：分析代码

从以下几个维度分析：

#### 功能正确性
- 改动是否解决了预期的问题？
- 是否有边界情况没处理？
- 逻辑是否清晰？

#### 代码质量
- 命名是否清晰？（变量、函数、文件名）
- 是否有重复代码？
- 函数是否职责单一？

#### 潜在问题
- 是否有 bug 风险？（空指针、边界条件、异步问题）
- 是否有性能问题？
- 是否有安全风险？

#### 与项目风格的契合度
- 是否遵循项目的代码规范？
- 是否符合项目的架构设计？

### 第三步：输出报告

```
## Code Review 报告

### 概述
（一句话描述这个 PR/改动做了什么）

### 优点
- ✅ 改动点 1
- ✅ 改动点 2

### 问题与建议

#### 🔴 必须修复
| 文件 | 问题 | 建议 |
|------|------|------|
| src/a.ts | 缺少空值检查 | 添加 if (!x) return |

#### 🟡 建议改进
| 文件 | 问题 | 建议 |
|------|------|------|
| src/b.ts | 函数过长 | 拆分为子函数 |

#### ⚪ 讨论项
- 这个实现方式和现有的 X 有点不同，要统一吗？

### 总结
整体评价：优秀 / 良好 / 需要修改后才能合并
```

## 审查原则
- **建设性**：指出问题的同时给出解决方案
- **优先级**：先说必须修复的，再说建议改进的
- **具体化**：每个问题都要精确到文件和行号
