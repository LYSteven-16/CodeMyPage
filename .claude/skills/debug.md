# Debug 技能 — 调试诊断

当用户说"debug""调试""有问题""报错""出错了""不对"时触发。

## 执行步骤

### 第一步：收集信息

1. **用户描述的问题**
   - 期望行为是什么？
   - 实际行为是什么？
   - 什么时候出现？（页面加载时/点击后/特定操作）

2. **错误信息**
   - 控制台报错（如果有）
   - 网络请求失败（如果有）
   - 浏览器崩溃（如果有）

3. **复现步骤**
   - 能否稳定复现？
   - 最简单的复现方式是什么？

### 第二步：定位问题

#### 如果是构建/运行时错误
```bash
npm run build 2>&1
# 或
npm run dev 2>&1
```
读取错误输出，定位到具体文件和行号。

#### 如果是运行时 bug
1. **搜索相关代码**：
```bash
# 搜索关键词
grep -rn "关键词" src/

# 查看相关文件
glob **/*.{ts,js,json}
```

2. **追踪数据流**：
   - 找到问题出现的入口
   - 追踪状态变化
   - 找出异常点

#### 如果是浏览器端问题
1. 确认问题出现的浏览器和环境
2. 检查是否有控制台错误
3. 检查网络请求（XHR/Fetch）

### 第三步：分析根因

在找到问题后，**不要急着修复**，先确认根因：

- 这个问题是真的 bug 还是预期行为？
- 是这个问题还是其他问题的表象？
- 修复会不会引入新问题？

### 第四步：输出诊断报告

```
## Debug 诊断报告

### 问题描述
（用户描述的问题）

### 信息收集
- 错误类型：构建错误 / 运行时错误 / 逻辑错误
- 错误信息：（具体的错误输出）
- 复现方式：（能否稳定复现）

### 根因分析
- 文件：`src/xxx.ts`
- 位置：第 N 行
- 原因：（解释为什么出问题）

### 建议修复方案
1. 修改 `src/xxx.ts` 第 N 行：...
2. 或者：...

### 风险评估
- 修复风险：低 / 中 / 高
- 可能影响：（关联的功能）
```

## 常见问题速查

| 问题 | 排查方向 |
|------|---------|
| 构建失败 | 检查 TS 类型、导入路径、依赖安装 |
| 页面空白 | 检查 main.ts 入口、DOM 挂载点、console 错误 |
| 数据不更新 | 检查 state 变更、events 触发、render 调用 |
| 样式错乱 | 检查 class 名、UnoCSS 配置、样式覆盖 |
| 请求失败 | 检查 URL、CORS、网络请求参数 |
